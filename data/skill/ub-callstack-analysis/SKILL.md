# ub-callstack-analysis

## 目标

分析 UB 组件源码中的函数调用关系，生成有向调用图。

适用场景：

- 追踪 UB 相关崩溃路径
- 评估跨组件函数调用依赖

## 支持组件

- `ubsocket`
- `umq`
- `liburma`
- `libudma`

## 输入

- `component`：必选，且必须是支持组件之一
- `source_path`：必选，组件源码根路径
- `compile_commands_path`：可选，`compile_commands.json` 文件路径

### 路径优先级规则

1. 明确只扫描 `{.h/.hpp/.c/.cc/.cxx/.cpp}` 文件

### 编译数据库优先规则

1. 若提供 `compile_commands_path`，优先使用其指向的 `compile_commands.json` 中的编译单元与编译参数进行解析
2. 若未提供 `compile_commands_path` 且 `source_path` 下存在 `compile_commands.json`，优先使用该文件进行解析
3. 当 `compile_commands.json` 覆盖不全或存在缺失时，再按组件源码根路径普通扫描

## 输出

输出目录：`/var/witty-ub/callstack-analysis/<component>`

输出文件：

- `callstack.json`

### callstack.json 结构

```json
{
  "nodes": [
    {
      "id": "umq::umq_send@send.c",
      "name": "umq_send",
      "component": "umq",
      "file": "send.c"
    }
  ],
  "edges": [
    {
      "src": "umq::rpc_entry@rpc.c",
      "dst": "umq::umq_send@send.c",
      "confidence": "confirmed"
    }
  ]
}
```

字段规则：

- `nodes`：函数节点列表
- `edges`：调用边列表，方向固定为 `caller(src) -> callee(dst)`
- `nodes[].id`：全图唯一，推荐格式 `组件::函数@文件`
- `nodes[].name`：裸函数名
- `nodes[].file`：裸文件名
- `edges[].src/dst`：必须引用已存在的 `nodes[].id`
- `edges[].confidence`：仅允许 `confirmed` 或 `inferred`

补充约束：

- 不得输出悬空边（引用不存在节点）

## 分析规则

### 调用关系覆盖范围

- 直接调用
  - 普通函数调用
  - 普通结构体/类成员函数调用
  - 虚函数分发调用
- 间接调用
  - 函数指针分发调用
  - 回调函数

### 函数定义识别主通道规则（AST 优先）

1. 函数定义识别必须优先使用编译器前端 AST（如 clang/libclang）完成，禁止仅依赖正则表达式直接产出最终 `nodes`
2. 仅当符号可对应到“函数定义语法节点”时，才允许创建函数节点
3. 若 `compile_commands.json` 可用，应以其编译参数驱动 AST 解析；覆盖不全时再按源码路径补扫
4. 文本回退仅可作为兜底，且回退产出的节点必须通过“回退节点二次校验规则”后方可写入 `callstack.json`

### 函数声明/定义识别补充规则

为避免遗漏返回值为指针/引用的函数，函数名提取必须覆盖以下声明或定义形态：

1. 普通返回：`Type func(...)`
2. 指针返回（符号贴函数名）：`Type *func(...)`
3. 引用返回（符号贴函数名）：`Type &func(...)`
4. 多级指针/引用组合：`Type **func(...)`、`Type *&func(...)`、`Type &*func(...)`（按实际语言合法性解析）

约束：

1. `*`、`&` 归属于返回类型修饰，不得因此跳过后续函数名识别
2. 无论 `*`、`&` 与函数名之间有无空格（如 `*func`、`* func`），也必须提取 `func` 为函数名
3. 同时适配声明与定义（含类外定义、命名空间限定名、模板上下文中的普通函数）

### 回退节点二次校验规则（防伪函数）

当 AST 解析失败或覆盖不足，允许文本回退；但每个候选函数节点必须同时满足：

1. 候选定义位于文件顶层作用域（非函数体内部、非局部语句块）
2. 候选签名前缀不含赋值初始化语义（如 `=`、复合字面量、结构体静态初始化片段）
3. 候选名满足标识符规则 `[A-Za-z_][A-Za-z0-9_]*`
4. 若函数名长度为 1（如 `f`、`i`），必须在同文件可回溯到完整“返回类型 + 函数名 + 参数列表 + 函数体”定义；否则禁止建节点
5. 禁止将控制流语句头、宏展开残片、属性注解残片识别为函数定义

### 公共 API 声明-定义一致性补漏规则

1. 当组件存在对外 API 头文件时，应先抽取其中声明的公共函数集合，再与源码中的函数定义集合做一致性比对
2. 若某公共函数已在组件源码中存在可定位定义，但未进入 `nodes`，必须补入对应节点
3. 对补入节点，应按既有规则继续补建其可解析调用边；若暂无法解析边，至少保留函数节点，禁止静默丢失
4. 若声明存在但在当前扫描范围内无可定位定义，可不创建节点，并在分析备注中标记为“仅声明未定义（当前范围）”

### 同名函数匹配规则

当上下游存在同名函数时，按以下顺序匹配：

1. 优先匹配调用点所在文件内定义
2. 若未命中，按 `include` 头文件展开匹配
3. 若仍不唯一或无法确定，同时纳入所有候选同名函数

### 运行期分发调用解析规则

遇到函数指针或虚函数调用时：

1. 追踪函数指针赋值来源或子类虚函数实现
2. 若调用对象来自 getter 返回值，追踪 getter 返回的静态表对象并展开字段映射

#### 结构体函数指针表（ops）解析规则

针对 `obj->ops->fp(...)`、`obj.ops.fp(...)` 这类调用，按以下顺序解析：

1. 识别函数指针字段类型（如 `int (*umq_tp_poll)(...)`），并锁定被调用字段名（如 `umq_tp_poll`）
2. 解析该字段在结构体静态初始化中的赋值（如 `.umq_tp_poll = umq_tp_xxx_poll`）
3. 若调用对象来自 `xxx_ops_get()` 返回值，需将返回的静态 ops 对象与调用点对象建立映射
4. 若存在跨对象传递（如 `a->ops = b->ops`），需沿赋值链继续追踪直至静态定义或可确认来源
5. 若同一字段存在多实现（不同 mode/目录/编译分支），所有候选实现都应作为调用点下游边输出，`confidence` 标记为 `inferred`
6. 仅当调用点可唯一映射到单一实现时，方可标记为 `confirmed`
7. 若实现来自动态加载（如 `dlsym/load_symbol`）无法静态唯一解析，至少保留静态可见候选实现并标记为 `inferred`

### 自环边约束规则

1. 除自身递归调用外，禁止出现自环边（`edges[].src == edges[].dst`）
2. 仅当调用点可证实为函数直接或间接自递归时，允许保留自环边
3. 因非自身递归调用产生的自环，视为非法自环并删除

### 重复边约束规则

1. 若一个函数多次调用另一个函数，应视为存在上下游关系并聚合成一条边

## 过滤规则

### 非函数符号过滤规则

为避免将非函数符号误识别为函数节点，必须执行以下过滤：

1. 禁止将结构体/类/联合体名称识别为函数（如 `struct xxx`、`class xxx`、`union xxx` 中的类型名）
2. 禁止将命名空间、类型别名、模板参数、变量名、字段名识别为函数
3. 禁止将编译器扩展属性或注解识别为函数（如 `__attribute__(...)`、`__declspec(...)`、`[[...]]`）
4. 禁止将宏调用识别为函数，禁止创建宏节点和直接与宏相关的调用边
5. 禁止将构造函数初始化列表中的成员变量识别为函数
6. 仅当符号可与“函数定义”或“函数调用表达式”建立语法对应关系时，才允许创建函数节点与调用边

### 外部函数过滤规则

1. 对系统调用、C 标准库函数、glibc 函数、C++ `std` 命名空间函数进行统一过滤，不创建对应 `nodes`，且不输出指向这些函数的 `edges`

### 测试语义过滤规则

1. 测试函数不纳入调用栈分析
2. 若函数名、文件名或目录名具有“测试”语义，对应节点与调用边均应过滤
3. “测试”语义可按常见关键词识别（如 `test`、`example`、`unittest`、`mock`、`benchmark`、`perftest`）

## 置信度判定

- `confirmed`：可直接定位到唯一被调函数定义（或唯一可确认实现）
- `inferred`：存在多候选实现、跨宏/模板展开不完全、仅能依据赋值链或类型关系推断

## 步骤

### 1. 直接调用关系检索

1. 搜索所有函数名，得到函数列表（必须按“函数声明/定义识别补充规则”覆盖 `Type *func()`、`Type &func()` 等写法）
2. 以函数名为范围，按上述规则建立直接调用关系
3. 对于虚函数，将所有子类的同名虚函数都作为下游
4. 若组件存在公共 API 头文件，执行“声明集合 vs 定义集合”一致性比对，并对缺失节点执行补漏

### 2. 间接调用关系检索

1. 搜索所有函数指针，得到函数指针列表
2. 搜索这些函数指针被调用的位置，并用被调用函数名匹配步骤1得到的函数列表
3. 将匹配到的函数作为一次间接调用的上游
4. 搜索这些函数指针真正被赋值的位置，并用赋值函数名匹配步骤1得到的函数列表
5. 将被赋值的函数作为一次间接调用的下游

## 失败与回退策略

- 文件解析失败：跳过该文件并继续
- 无法唯一解析同名函数：保留所有候选边，标记为 `inferred`

## 质量自检清单

- 是否严格输出到 `/var/witty-ub/callstack-analysis/<component>/callstack.json`
- `nodes[].id` 是否全局唯一
- `nodes[].name` 存在相同项时，回检所在文件，避免重复检测和幻觉
- `edges[].src/dst` 是否全部可回溯到节点
- `file` 字段是否均为裸文件名
- `confidence` 是否仅为 `confirmed/inferred`
- 若组件存在公共 API 头文件，声明函数与已输出节点是否完成一致性比对，且无“有定义但缺节点”的静默遗漏
- 是否已执行“AST 优先 + 回退二次校验”
- 是否存在单字符函数名节点；若存在，是否具备可追溯定义证据
- 是否存在疑似伪函数（变量名/字段名/宏名）进入 `nodes`
